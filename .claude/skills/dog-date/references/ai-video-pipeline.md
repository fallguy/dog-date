# AI video pipeline

Read this when you're touching `generate-dog-video`, `fal-poll`, the video status banner / preview modal, the seed-demo-videos script, or anything that involves Fal.ai.

## What gets generated

A 5-second 9:16 video of the user's dog in a fun viral scenario (pilot, chef, surfer, etc.) generated by Fal.ai's Veo 3.1 Lite image-to-video model. Roughly $0.30 per clip. The dog's primary photo is the seed image; the prompt templates the breed name in (e.g. "A Golden Retriever dog wearing aviator goggles ...").

## End-to-end flow

```
[client]                  [generate-dog-video]                   [Fal queue]
   │                              │                                  │
   │ POST /generate-dog-video ───▶│                                  │
   │ { dog_id, prompt, scenario } │                                  │
   │                              │ verify auth, ownership, photo,  │
   │                              │ cost cap (MAX_VIDEOS_PER_DOG)   │
   │                              │ download photo, base64, submit  │
   │                              │ POST /fal-ai/veo3.1/lite/...    │
   │                              │ ────────────────────────────▶  │
   │                              │  ◀── { request_id }            │
   │                              │ INSERT video_jobs               │
   │                              │ UPDATE dogs.ai_video_status     │
   │                              │   = 'pending'                   │
   │ ◀── { video_job_id, ... }    │                                  │
   │                              │                                  │
[useVideoPoller, every 5s]                                            │
   │ POST /fal-poll ──────────▶ [fal-poll]                            │
   │   { video_job_id }            │ verify ownership                │
   │                               │ GET /fal-ai/veo3.1/requests/    │
   │                               │   <id>/status                   │
   │                               │ ────────────────────────────▶  │
   │                               │  ◀── { status: COMPLETED }     │
   │                               │ GET /fal-ai/veo3.1/requests/<id>│
   │                               │  ◀── { video: { url } }        │
   │                               │ UPDATE dogs.ai_video_url +     │
   │                               │   ai_video_status = 'ready'    │
   │ ◀── { status: 'ready', url } │                                  │
[VideoStatusBanner flips: 'generating ✶' → 'ready']
[user taps banner → VideoPreviewModal]
```

## Two URL shapes — easy to get wrong

- **Submit:** `https://queue.fal.run/fal-ai/veo3.1/lite/image-to-video`
- **Status / result:** `https://queue.fal.run/fal-ai/veo3.1/requests/<request_id>[/status]`

The status URL drops `/lite/image-to-video` — it lives at the **parent app slug** `fal-ai/veo3.1`, not the operation slug. We hit a 405 on the wrong URL and the job silently sat in `pending` forever before catching this. The two URLs are hardcoded as `FAL_MODEL` (in `generate-dog-video`) and `FAL_APP` (in `fal-poll`). **If you change models, update both.**

## Aspect ratio constraints

Veo 3.1 Lite accepts `auto`, `16:9`, or `9:16`. Hardcoded to `9:16` for the swipe-card aesthetic. Submit returns a `request_id` even for invalid params, but the *result* endpoint then 422s — silent failure mode. If you change the aspect ratio, exercise it on a real dog before merging.

## Cost cap behavior

`MAX_VIDEOS_PER_DOG` env var (default 3) limits how many videos a single dog can have generated. The cap counts only `pending` and `ready` jobs; `failed` doesn't count toward the cap. This is deliberate — if our bug failed a job, we shouldn't burn one of the user's tries.

If a user hits the cap in dev, bump the env var: edit `supabase/functions/.env` (gitignored — contains the live FAL_KEY too) and restart `supabase functions serve`.

## Local Storage isn't reachable from Fal

`generate-dog-video` doesn't pass the photo URL to Fal directly. Photos live at `http://<LAN-IP>:54321/storage/v1/...` — Fal's servers can't see your LAN. So the edge function:

1. `fetch(dog.primary_photo_url)` (server-side, has access)
2. Read response body as `ArrayBuffer`
3. Base64-encode → `data:<content-type>;base64,<...>`
4. POST that data URI to Fal as `image_url`

When the project goes to cloud Supabase, photos will live at a public Supabase Storage URL that Fal CAN see — at that point, you can switch to passing the URL directly and skip the data-URI step. Saves ~50ms and reduces request size from ~150KB base64 to a string.

## Polling, not webhooks

Fal supports webhooks but they need a public callback URL — the local dev server can't receive them. We poll instead.

`lib/hooks/useVideoPoller.ts` is mounted globally in `_layout.tsx`. It watches `useMyDog`. When `dog.ai_video_status === 'pending'`, it calls `supabase.functions.invoke('fal-poll', { body: { video_job_id } })` every 5s. When the status flips to `ready` or `failed`, it invalidates the `['my-dog', userId]` query and stops polling.

Cloud deploy time: switch to webhooks for efficiency. Add `?fal_webhook=https://...` to the submit URL; Fal POSTs the result back instead of you polling. Until then, 5s polling is fine for local dev.

## Edge function structure

Both edge functions follow the same pattern (mirror it for new functions):

```ts
import { createClient } from 'jsr:@supabase/supabase-js@2';
import { corsHeaders, jsonResponse } from '../_shared/cors.ts';

const FAL_KEY = Deno.env.get('FAL_KEY');
// ...

Deno.serve(async (req) => {
  if (req.method === 'OPTIONS') {
    return new Response('ok', { headers: corsHeaders });
  }

  try {
    // 1. Validate FAL_KEY env var present
    // 2. Read Authorization header, return 401 if missing
    // 3. Build a userClient (with auth header) and an adminClient (service role)
    // 4. Call userClient.auth.getUser() to validate the JWT
    // 5. Read + validate body
    // 6. Verify resource ownership via the userClient (RLS applies)
    // 7. Do the actual work using adminClient (bypasses RLS for the writes that need it)
    // 8. Return jsonResponse(...)
  } catch (e) {
    return jsonResponse({ error: (e as Error).message }, 500);
  }
});
```

Why two clients: the user client respects RLS (so `select` queries can't read other users' data), but the admin client is needed for cross-cutting writes (insert into `video_jobs` with a service role, update `dogs.ai_video_url` even if the user doesn't have that column in their RLS write policy). Keep ownership checks on the user client; do the privileged write on the admin client *after* verifying.

## seed-demo-videos.mjs — the demo bootstrapper

`scripts/seed-demo-videos.mjs` generates AI videos for the first 3 demo dogs (Biscuit/Pickle/Juno) so the demo deck shows real AI videos to first-time viewers. ~$0.90 total.

It's idempotent: re-running skips dogs that already have `ai_video_status = 'ready'`.

It bypasses the edge function entirely (the demo dogs aren't owned by your auth.users) and submits directly to Fal with the `FAL_KEY` from `supabase/functions/.env`. After receiving each video URL, it updates the dog row directly via `psql`.

Known gotcha: the script's `psql` calls use `$$` as the dollar-quote tag, which **bash expands to the shell PID** (e.g. `$$` → `68865`). The `video_jobs` INSERTs fail because the prompt gets prefixed with the PID. The dog UPDATEs succeed (no `$$` in those statements), so the AI videos still land — just no `video_jobs` rows for these. If you ever fix the script properly, change the dollar-quote tag to `$prompt$` or any unique identifier that bash doesn't expand.

## Status banner state machine

`components/VideoStatusBanner.tsx` reads `useMyDog`, the latest `video_jobs` row, and the `lastSeenVideoUrl` from `lib/notifications-store.ts`. States:

| `ai_video_status` | `lastSeenVideoUrl === ai_video_url`? | Banner |
|---|---|---|
| `idle` | n/a | hidden |
| `pending` | n/a | "{name} is becoming a star…" + mm:ss counter |
| `ready` | yes | hidden (already acknowledged) |
| `ready` | no | "{name}'s video is ready." + Preview link + dismiss × |
| `failed` | n/a | "{name}'s video didn't generate." + Retry link |

Tapping `Preview →` opens `VideoPreviewModal` and calls `markSeen(url)` so the banner doesn't re-trigger. Tapping `×` also calls `markSeen`.

The mm:ss timer reads `started_at` from the latest pending `video_jobs` row. If you find the timer ticking forever (>5 min), there's a stale `pending` job somewhere that fal-poll never resolved. Studio (http://localhost:54323) shows the table; check `fal_request_id` and curl the Fal status URL with the FAL_KEY to see what Fal thinks. Common cause: wrong status URL (see "Two URL shapes" above).

## "COMPLETED" doesn't mean "succeeded"

Fal's status API returns `COMPLETED` for both successful generations and failures-after-completion (e.g. safety-filter rejection, `no_media_generated`). The status endpoint says COMPLETED; the result endpoint returns a 4xx with a `detail` body. **`fal-poll` must treat "COMPLETED + no `result.video.url`" as a failure** — write `ai_video_status = 'failed'` to the dog row, `status = 'failed'` to the job row, and capture the error detail. If you don't, the client poller spins forever (~12 hits per minute) on a doomed endpoint, the banner sits at "generating", and the user never gets a Retry option.

A regression of this exact bug already cost an 18-minute live wait + ~216 wasted Fal API calls. The current `fal-poll` handles it correctly; if you ever rewrite it, preserve that behavior.

A common reason for `no_media_generated`: the user-typed breed is unusual or hits a safety filter. The prompt template substitutes the breed verbatim — e.g. a breed of `"Badass"` produces `"A Badass dog ..."` which Veo refused. Consider sanitizing or allow-listing breeds at the onboarding layer if this becomes common.

## Tests worth having

The video pipeline is exactly the layer where a regression costs real money, so tests are valuable:

- **Submit-side validation** — invalid prompt length, missing dog, not-your-dog, cost-cap reached. All testable without burning Fal credits.
- **Cost cap counts only pending/ready** — set up 3 failed jobs and verify the user can still submit.
- **Status URL formation** — a unit test that builds the URL string from a fake request_id and asserts it matches `https://queue.fal.run/fal-ai/veo3.1/requests/<id>/status`. Prevents regression of the URL-shape bug.
- **fal-poll status mapping** — IN_QUEUE → pending, IN_PROGRESS → pending, COMPLETED → ready, anything else → failed. Mock the Fal HTTP layer (Deno test fetch interception); don't actually hit Fal.

The `__tests__/edge-functions/` directory has skeletons. Flesh them out before the next big change to the pipeline.

## What "changing models" looks like

If you want to swap to a different Fal model (e.g., a newer Veo, Minimax Hailuo, Kling):

1. Update `FAL_MODEL` in `generate-dog-video/index.ts`.
2. Update `FAL_APP` in `fal-poll/index.ts` to match the parent app slug. **Verify the URL split** — some Fal models have submit and status at the same path; others split. Test on one dog before rolling out.
3. Verify the new model's accepted aspect ratios; update the hardcoded `aspect_ratio` if needed.
4. Verify pricing — update the comment in `_layout.tsx` / project docs.
5. Test the full submit → poll → ready cycle on a real photo before merging.
6. The seed-demo-videos script doesn't reference `FAL_MODEL` from the edge function; update its constants too.
